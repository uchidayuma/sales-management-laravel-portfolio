# EKS Fargate移行とセキュリティ強化時のトラブルシューティングまとめ

このドキュメントは、EKS (EC2) から AWS Fargate に移行し、コンテナのセキュリティを本番レベルに引き上げた際に発生した問題とその解決プロセス（思考プロセス）をまとめたものです。
KubernetesやAWSを実務で運用する際に、非常に直面しやすい**「インフラとアプリ設定の隙間に落ちる落とし穴」**を学ぶための教材として活用してください。

---

## 1. データベースへの接続タイムアウト (Security Groupの罠)

### 🚨 事象
コンテナログに `PDOException: Connection refused` や `Operation timed out` が記録され、LaravelからAWS RDS(MySQL) や Elasticache(Redis) へ接続できず、500エラーが発生しました。

### 🔍 原因
EC2環境では、Node (EC2インスタンス) にアタッチされた「EC2用のセキュリティグループ」からのアクセスをDB側で許可していました。
しかし、**Fargateに移行すると、Pod自体が直接VPC内に配置され、「EKSクラスタのセキュリティグループ」を使って通信する**ようにネットワーク経路が変わりました。これにより、従来のポート開放ルールにマッチせず、通信がファイアウォール（セキュリティグループ）でブロックされていました。

### ✅ 解決策
TerraformのDBリソース (`database.tf`, `cache.tf`) における Ingress（受信）ルールを修正し、EKSクラスタのセキュリティグループ (`cluster_primary_security_group_id` など) からの通信を許可しました。

> **【講座のポイント💡】**
> コンピューティング基盤（EC2 -> Fargate等）を変更した際は、単にアプリを乗せせ替えるだけでなく、「ネットワーク経路（どのセキュリティグループが適用されるか）」の再確認が必須です。

## 2. 強固なセキュリティ設定とLaravelの相性問題 (readOnlyRootFilesystem)

### 🚨 事象
DB接続は復旧したものの、依然として500エラーが発生。ログを見ると「`/var/www/storage/logs/...` に書き込めません (Read-only file system)」というエラーが出ていました。

### 🔍 原因
コンテナをマルウェア感染や改ざんから守るため、**コンテナ全体のファイルシステムを書き込み不可にする最強の保護設定** (`readOnlyRootFilesystem: true`) を有効にしていました。
しかし、Laravel（および一部のプラグイン）はリクエストを処理するたびに `storage/logs` や `storage/framework/cache/` へ一時ファイルを書き込む必要があるため、書き込みがOSレベルでブロックされアプリが停止しました。

### ✅ 解決策
Kubernetesの機能である `emptyDir` (コンテナのメモリ上の一時領域) を活用しました。
システム全体は読み取り専用で保護したまま、**Laravelがどうしても書き込みを必要とする特定のパスだけ**に、この揮発性の一時領域を上書きマウント（volumeMounts）しました。

**修正したマニフェスト (`04-deployment.yaml`)：**
```yaml
          volumeMounts:
            - name: tmp-volume
              mountPath: /tmp
            - name: tmp-volume
              mountPath: /var/www/storage/logs
            - name: tmp-volume
              mountPath: /var/www/storage/framework/cache/data
            # ... 他に必要なディレクトリも同様に追加
```

> **【講座のポイント💡】**
> 「セキュリティを厳しくするとアプリが動かなくなる」という実務あるあるの典型例です。アプリの仕様を理解し、例外的に許可するパスを正確に見極める設計力が求められます。

## 3. ALBのヘルスチェックエラー（302リダイレクトの許容）

### 🚨 事象
AWS Load Balancer ControllerでALBを自動構築したものの、ターゲットグループでPodが「Unhealthy（異常）」と判定されました。その結果、ブラウザからアクセスしてもALBから `ERR_TIMED_OUT` や `502 Bad Gateway` が返ってきてしまいました。

### 🔍 原因
ALBは定期的にコンテナへヘルスチェック（死活監視のリクエスト）を送信しますが、デフォルトでは `200 OK` のみを「健康」と判定します。
しかし、Laravelは未ログイン状態でルートパス (`/`) にアクセスされると、ログイン画面 (`/login`) へリダイレクトするため、**`302 Found`** を返します。ALBは「200じゃないからこのコンテナは壊れている！」と勘違いし、通信を遮断してしまいました。

### ✅ 解決策
Ingressのマニフェスト (`06-ingress.yaml`) にアノテーションを追加し、AWS側に「302が返ってきても 정상（正常）だとみなしてね」と教え込みました。

```yaml
    # Laravelのリダイレクト(302)をヘルスチェック成功として許容する
    alb.ingress.kubernetes.io/success-codes: "200,302"
```

> **【講座のポイント💡】**
> Webフレームワーク特有の動作（リダイレクト等）と、クラウドインフラ(ALB)の死活監視の仕様のズレは非常によく起きるトラブルです。監視設定はアプリケーションの実装に寄り添ってチューニングする必要があります。

## 4. 環境変数の安全な注入 (Kubernetes Secretの実運用)

### 🚨 事象と課題
データベースのパスワードなどの「機密情報」をGitHubで管理するマニフェストファイル (`04-deployment.yaml`) に直接記述するのは、甚大なセキュリティ事故（情報漏洩）につながります。

### ✅ 解決策
AWS Secrets Manager と Kubernetes の `Secret` リソースを組み合わせたデプロイフローを構築しました。

1. 機密情報はAWS Secrets Managerで一元管理（Terraformでランダム生成して格納）。
2. デプロイスクリプト (`deploy-secrets.sh`) を実行した瞬間にAPIでパスワードを取得。
3. そのままK8sに `sales-app-secrets` という暗号化リソースとして作成。
4. Podからは `envFrom` を使って、安全にコンテナの環境変数として読み込む構成にしました。

> **【講座のポイント💡】**
> 「コード（Git）にはパスワードを含めず、デプロイの瞬間に安全な保管庫から取得して注入する」という運用が、モダンなCI/CDパイプラインを構築する上でのベストプラクティスです。
